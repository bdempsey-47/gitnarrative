# gitnarrative

**LLM-powered CLI that reads git history and generates narrative markdown describing a project's evolution.**

![Python 3.11+](https://img.shields.io/badge/python-3.11%2B-blue)
![License](https://img.shields.io/badge/license-MIT-green)

## What is gitnarrative?

gitnarrative reads your git commit history, clusters commits into logical features, and uses Claude to generate narrative markdown explaining what was built and why. Think of it as **automated Architecture Decision Records from git history**.

It's built for developers onboarding onto unfamiliar codebases and teams that want living documentation that stays in sync with the code. Point it at any repo and get a structured narrative of its evolution.

## Sample Output

Running `gitnarrative narrate --repo ./weather-app --since 2024-06-01` produces a synthesized narrative like this:

```markdown
# weather-app — Project Narrative

> Auto-generated by gitnarrative on 2024-08-15 14:32
> Commit range: 2024-06-01 to now
> 47 commits across 23 files

---

## Project Overview

A weather forecast web app that resolves user coordinates to grid points via
the NWS API, fetches 7-day forecasts, and layers NEXRAD radar data onto an
interactive Leaflet map. Built with Python/Flask, Redis for caching, and
client-side tile compositing to avoid server costs.

## Key Features & Milestones

### Location-Based Forecast System (Jun 3–18, 2024)

Built a geolocation-aware forecast pipeline with 30-minute Redis caching to
stay within NWS rate limits (500 req/hr). Falls back to stale cache on API
timeout rather than showing an empty state.

**Key Decisions:**
- Chose NWS API over OpenWeatherMap to avoid per-call billing
- Fallback to stale cache on timeout rather than empty state

**Primary Files:** `forecast/nws_client.py`, `forecast/cache.py`, `forecast/resolver.py`

### Interactive Radar Map (Jun 20 – Jul 1, 2024)

Leaflet-based radar overlay compositing NEXRAD tiles client-side. Users can
scrub through the last 2 hours of radar frames.

**Key Decisions:**
- Used Iowa State Mesonet for free NEXRAD tile access
- Client-side compositing avoids server-side image processing costs

**Primary Files:** `static/js/radar.js`, `templates/map.html`, `forecast/radar_tiles.py`

## Traps & Gotchas

- **NWS grid resolution is cached separately from forecasts** — Grid point
  lookups and forecast data have different staleness characteristics. Removing
  the two-layer cache will cause subtle freshness bugs. (`forecast/cache.py`)

## Hotspots

- **`forecast/nws_client.py`** (9 commits) — Rapid iteration adapting to
  undocumented NWS API quirks (rate limiting, inconsistent error responses).

---

*Generated by gitnarrative v0.1.0*
```

Use `--no-synthesize` to get verbose per-feature output instead (one section per cluster with full phases, decisions, and a minor changes table).

## How It Works

gitnarrative runs a six-stage pipeline:

1. **git reader** -- Extracts commits (SHA, date, author, message, file stats) via GitPython
2. **clusterer** -- Groups commits by file co-occurrence and time proximity using heuristics (no LLM)
3. **classifier** -- Labels each cluster as major or minor
4. **narrator** -- Sends major clusters to Claude Haiku, receives structured JSON (summary, phases, decisions, files)
5. **synthesizer** -- Sends all narrations + file hotspot data to Claude Sonnet, produces a concise onboarding document with project overview, key features, traps/gotchas, and hotspots
6. **output** -- Renders through Jinja2 templates into markdown

### Major vs Minor Classification

Not every cluster needs a full LLM narrative. gitnarrative classifies clusters to keep output focused:

- **Multi-commit clusters** are always **major** (they represent real features)
- **Single-commit clusters** are **minor** if they match any of:
  - Docs-only changes (`.md`, `.txt`, `.rst`, `LICENSE`, `docs/`)
  - CI/config-only changes (`.yml`, `.yaml`, `.toml`, `.cfg`, `Dockerfile`, `.dockerignore`, `.gitignore`, `.github/`)
  - Commit prefix is `chore:`, `docs:`, `ci:`, `style:`, or `build:`
  - Merge commits or empty commits

Major clusters get full LLM narration. Minor clusters are excluded from the synthesized output (or rendered as a compact table in `--no-synthesize` mode).

The `--detail` flag controls this behavior:

| Value | Behavior |
|-------|----------|
| `full` | Narrate every cluster (no minor classification) |
| `significant` | Skip minor clusters from narration |
| `auto` (default) | Use `significant` when >20 clusters, `full` otherwise |

## Getting Started

### Prerequisites

- Python 3.11 or later
- An [Anthropic API key](https://console.anthropic.com/)

### Install

```bash
pip install git+https://github.com/bdempsey-47/gitnarrative.git
```

Or for development:

```bash
git clone https://github.com/bdempsey-47/gitnarrative.git
cd gitnarrative
pip install -e .
```

### API Key Setup

Store your key securely in the system keyring:

```bash
gitnarrative config set-key
```

Or set it as an environment variable:

```bash
export ANTHROPIC_API_KEY=sk-ant-...
```

The environment variable is checked first; the keyring is used as fallback.

## Usage

### Basic

```bash
gitnarrative narrate --repo ./my-project --since 2024-01-01
```

Output is written to `.gitnarrative/narrative.md` inside the target repo.

### Dry Run

Test the pipeline (stages 1-3) without making any LLM calls:

```bash
gitnarrative narrate --repo ./my-project --since 2024-01-01 --dry-run
```

### CLI Options

| Option | Default | Description |
|--------|---------|-------------|
| `--repo` | `.` | Path to the git repository to analyze |
| `--since` | None | Start date (`YYYY-MM-DD` or `YYYY-MM-DDTHH:MM:SS`) |
| `--until` | None | End date (same formats) |
| `--model` | `claude-haiku-4-5-20251001` | Claude model for narration (synthesis uses Sonnet) |
| `--max-commits` | `500` | Maximum commits to process |
| `--detail` | `auto` | Detail level: `full`, `significant`, or `auto` |
| `--no-synthesize` | off | Skip synthesis; output verbose per-feature narrations |
| `--dry-run` | off | Show clusters without calling the LLM |

## Using as a Claude Code Skill

You can register gitnarrative as a Claude Code slash command so Claude can run it to bootstrap its understanding of a repo.

Create a file at `.claude/commands/narrate.md` in any project:

```markdown
---
description: Generate a narrative of this repo's git history
---

Run gitnarrative against the current repository to produce a structured narrative
of its evolution. Use the output to understand what was built, in what order,
and what key decisions were made.

Steps:
1. Run: `gitnarrative narrate --repo . --since 2024-01-01`
2. Read the output at `.gitnarrative/narrative.md`
3. Use the narrative to inform your understanding of the codebase architecture
   and design decisions
```

Then use `/narrate` in Claude Code to generate and ingest the narrative.
